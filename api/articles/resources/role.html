Роль/Функция (Role)

Отображает функциональное назначение сотрудников, которые к ней прикреплены.

Note

Функция и роль в контексте системы это синоним.

Свойства

Свойство Тип Домен Описание
claims RoleClaim[]? Клеимы функции, определяют набор разрешений
users User2Role[]? Сотрудники функции
order int >0 Номер функции по порядку среди дочерних относительно родительской
colorHex string HEX format Цвет карточки
calculated RoleCalculated? Содержит рассчитанные данные. Например список действий над функцией
🆕 positions RolePosition[] Должности функции
Warning

Чтобы сервер сам рассчитал order при запросах создания/переноса, должно выполняться условие order == 0.

🆕 Доп. параметры запросов ресурса

  • nestedLevel - количество уровней вложенности в модели для свойства .childRoles
  • includeActions - включить действия в модель (нужен как минимум nestedLevel=1)
  • includeClaims - включить клеимы в модель

Дополнительные запросы

  • GET/ api/v1/role/root - получение главной функции оргструктуры. Запрос поддерживает доп. параметры

RoleCalculated

Модель, в которой содержатся расчитанные данные. Нельзя изменять (точнее бессмысленно)

Свойство Тип Домен Описание
actions RoleAction[]? Действия над функцией
Note

🆕 RoleCalculated для функции можно получить по роуту GET/ {id}/calculated контроллера `Role.

Сотрудники функции

Данные о сотрудниках хранятся в свойстве users в виде коллекции модели User2Role

🆕 Клеимы функции

Данные о разрешениях хранятся в свойстве claims в виде коллекции модели RoleClaim

Для управления разрешениями функциями в интерфейсе выводится вся структура, полученная из запроса разрешений, где каждому элементу модели Permission сопоставляется RoleClaim из свойства .claims модели Role.

Warning

Сопоставление происходит по свойствам permission.key и roleClaim.type. Если сопоставить Permission к RoleClaim не удается - отображать отключенный свитч, иначе - вкл.

При переключении свитча разрешения - проводить соответствующие действия над свойством role.claims

Изменение порядка дочерних функций

Происходит при помощи запроса POST/ api/v1/{id}/order, где {id} - Id родительской функции по отношению к payload, переданному в body запроса.
