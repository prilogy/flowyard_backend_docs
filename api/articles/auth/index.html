Аутентификация и авторизация

Стратегии аутентификации

URI Описание
api/{v}/authentication/credentials Аутентификация с email и паролем
api/{v}/authentication/refresh-token Аутентификация с рефреш токеном

Вход в приложение производится по стратегии с email и паролем. Далее, для восстановления сессии при повторном входе, аутентифкаиця проводится по рефреш токену.

Каждая из стратегий возвращает 2 токена:

  • Refresh token - токен для повторной аутентификации позднее
  • Access token (bearer, JWT) - токен для авторизации запросов

В результате работы любой из стратегий может быть два ответа сервера:

  • 200 - аутентификация успешна
  • 400 - аутентификация провалена (в таком случае, если аутентификация уже была осуществлена, - происходит разлогин)

Access token

  • Должен добавляться к каждому запросу из авторизованной части приложения, в хедер Authorization в виде значения Bearer {access-token}
  • Не хранится в локальном хранилище приложения

Любой запрос, в связи с авторизацией, может вернуть следущие коды

  • 401 - access token невалиден или не передан
  • 403 - у пользователя не хватает прав на выполнение запроса/действия
