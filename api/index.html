Введение

Сервер придерживается RESTful архитектуры, тем не менее выполняет не все спецификации, поэтому является REST-Like.

Авторизация проводится на основе схемы Bearer

Базовая архитектура

RESTful контроллер

Или контроллеры ресуров, работают на основе одного интерфейса, и имеют сделующие эндпоинты:

URI Тип запроса Действие
api/{v}/{resource} POST Создание
api/{v}/{resource}/{id} PUT Редактирование
api/{v}/{resource}/{id} PATCH Частичное редактирование
api/{v}/{resource}/{id} DELETE Удаление
api/{v}/{resource} GET Получение списка
api/{v}/{resource}/{id} GET Получение одного

Где параметры:

  • {v} - версия контроллера

  • {resource} - название ресурса

  • {id} - Id ресурса

Описание каждого контроллера и эндпоинта API доступно в Swagger UI

Негативные статус коды

Подробнее про статус коды авторизации здесь.

При получении негативных кодов (400, 403, 404) может быть возвращено 2 разных модели, в зависимости от режима работы сервера:

  • В дебаг режиме - объект вида { message }, где message - текст ошибки
  • В продакшен режиме - строка, т.е. все тело ответа будет строкой, которое можно вывести
Note

Режим работы сервера можно получить с помощью запроса GET/ api/v1/helper/info, где в ответе в свойстве .isDebugMode будет и содержаться ответ на этот вопрос

🆕 Валидация форм

При действиях Create, Update, Delete над ресурсом и ошибке валидации - сервер возвращает код 400 с моделью { errors, message }, где:

  • errors - список объектов модели { propertyName: string, errorMessage: string } (самые значимые свойства)
  • message - ошибка в виде текста, как и при остальных сценариях
Пример объекта из свойства errors, данные использовать по усмотрению:
{
    "propertyName": "ParentRoleId",
    "errorMessage": "Тут текст ошибки в локализованном виде",
    "attemptedValue": 0,
    "severity": 0,
    "errorCode": "AsyncPredicateValidator",
    "formattedMessagePlaceholderValues": {
        "PropertyName": "Parent Role Id",
        "PropertyValue": 0
    }
}
